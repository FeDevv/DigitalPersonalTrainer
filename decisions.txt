# Registro delle Decisioni Progettuali (DPT)

In questo file sono documentate le scelte architetturali non banali che richiedono una giustificazione tecnica specifica.

---

## 1. Separazione tra Domain Models e Creation DTO (Input Records)

**Data della decisione**: 28/04/2026

**Contesto**: Durante lo sviluppo dei moduli utente, è sorta la necessità di decidere se utilizzare i modelli di dominio (es. `Client`) anche per le fasi di input/creazione o creare oggetti dedicati.

**Decisione**: Si è scelto di mantenere una separazione netta tra l'oggetto che rappresenta l'entità nel sistema (**Model**) e l'oggetto che trasporta i dati per la creazione (**DTO/Request**).

**Razionale Tecnico**:
*   **Integrità dello Stato (Identity vs Intent)**: Un modello di dominio deve rappresentare un'entità con un'identità certa (ID). Durante la creazione, l'ID non esiste ancora. Usare il Model costringerebbe a usare ID fittizi (es. 0), violando il principio di consistenza dell'oggetto.
*   **Sicurezza (Isolation of Secrets)**: I modelli di dominio (es. `User`) circolano in vari layer dell'applicazione. Includere la password nel Model la esporrebbe a rischi di sicurezza accidentali (es. log o stampe). Il DTO invece trasporta la password solo fino al DAO e poi viene eliminato.
*   **Disaccoppiamento dell'Evoluzione**: I dati richiesti in fase di input possono differire drasticamente dai dati persistiti (es. un DTO potrebbe richiedere una "Conferma Password" che non esiste nel DB). La separazione permette alla UI e allo schema DB di evolvere indipendentemente.

## 2. Dependency Injection Ibrida e Gestione DAO

**Data della decisione**: 30/04/2026

**Contesto**: Necessità di coordinare l'interazione tra diversi moduli (es. il Proprietario che gestisce PT e Clienti) mantenendo l'architettura pulita e i pacchetti isolati.

**Decisione**: Si è adottata una strategia di **Dependency Injection Ibrida**:
1.  **DAO Locali**: Ogni LogicController istanzia direttamente i DAO appartenenti al proprio pacchetto (es. `OwnerLogicController` crea `OwnerDAO`).
2.  **DAO Esterni**: I DAO appartenenti a pacchetti diversi vengono iniettati tramite costruttore dall'Orchestrator centrale.

**Razionale Tecnico**:
*   **Encapsulation (Package-Private)**: Permette di mantenere i DAO locali con visibilità di pacchetto dove possibile, riducendo la superficie pubblica dei moduli.
*   **Coordinamento Pulito**: L'Orchestrator funge da "Registry" centrale, eliminando dipendenze circolari tra controller e garantendo che le risorse condivise siano gestite in un unico punto.
*   **Flessibilità e Testabilità**: Facilita il mocking dei DAO esterni durante i test e permette di sostituire le implementazioni dei componenti globali senza modificare la logica interna dei singoli pacchetti.

## 3. Fail-Fast Validation nei DTO (DTO Hardening)

**Data della decisione**: 30/04/2026

**Contesto**: Prevenire la circolazione di dati inconsistenti (es. email malformate, età non valide) tra i vari layer dell'applicazione.

**Decisione**: I record DTO devono implementare costruttori compatti che eseguono la validazione dei dati tramite `ValidationUtils` o logiche specifiche (es. controllo lunghezza CF, età minima).

**Razionale Tecnico**:
*   **Integrità Atomica**: Un DTO non valido non può esistere. Se i dati non superano il controllo, l'eccezione viene lanciata immediatamente all'istante della creazione.
*   **Riduzione del Boilerplate nei DAO**: I DAO possono assumere che i dati ricevuti tramite DTO siano già formalmente corretti, concentrandosi solo sulla persistenza.
*   **Feedback Immediato**: Permette al sistema di segnalare errori di input il più vicino possibile alla fonte (UI), prima di impegnare risorse del DB o del network.

## 4. Pattern di Selezione Guidata (Guided Selection UI)

**Data della decisione**: 30/04/2026

**Contesto**: Evitare che l'utente debba conoscere o inserire manualmente ID tecnici (es. ID Macchinario, ID PT) durante le operazioni di inserimento/assegnazione.

**Decisione**: Ogni operazione UI che richiede un ID esterno deve essere preceduta dalla visualizzazione dell'elenco degli elementi disponibili recuperati dal database.

**Razionale Tecnico**:
*   **UX (User Experience)**: Riduce il carico cognitivo dell'utente e previene errori di inserimento di ID inesistenti.
*   **Integrità Referenziale**: Minimizza i fallimenti lato DB dovuti a violazioni di chiavi esterne, catturando l'intento dell'utente in modo guidato.
*   **Coordinamento Logic-UI**: Il LogicController ha la responsabilità di recuperare i dati necessari e passarli alla UI per la visualizzazione prima di richiedere l'input.
